# 向量数据库选型对比

> 向量数据库是 RAG 系统的存储基石。选错向量库，后期迁移成本极高（数据量大、接口不兼容、运维工具不同）。本文对比主流方案的适用场景、性能特征和运维成本。

## 一、为什么需要专门的向量数据库

传统关系型数据库无法高效处理"找最相似的 N 条向量"这类查询。暴力扫描 100 万条 1024 维向量需要数秒，而 HNSW 索引能把延迟降到毫秒级。

向量数据库的核心能力：
1. 高维向量存储（通常 768-1536 维）
2. 近似最近邻搜索（ANN）
3. 与业务数据的关联查询（按 user_id、kb_id 等过滤）

## 二、主流方案对比

### 2.1 PostgreSQL + pgvector

PostgreSQL 的官方向量扩展，目前最流行的轻量级方案。

**核心特性**：
- 维度支持：最高 1600 维
- 索引类型：IVFFlat、HNSW（推荐）
- 距离度量：L2、内积、余弦
- 过滤能力：WHERE 子句（与业务字段一起过滤，强项）

**适用场景**：
- 已有 PostgreSQL 基础设施
- 数据量 < 1000 万向量
- 需要强一致性、事务、复杂过滤

**局限**：
- 超大规模（亿级向量）性能下降
- 不支持分片（需要 citus 等扩展）

### 2.2 Milvus

国产开源专用向量库，目前 GitHub star 数最多的方案。

**核心特性**：
- 多种索引：FLAT、IVF、HNSW、DiskANN
- 支持十亿级向量
- 分布式架构（计算存储分离）
- 丰富的 SDK（Python、Java、Go）

**适用场景**：
- 大规模生产环境
- 团队有专职运维
- 需要多副本、高可用

**局限**：
- 部署复杂（etcd + MinIO + Pulsar 多组件）
- 内存占用大
- 与业务数据隔离，需双写维护关联

### 2.3 Qdrant

Rust 编写的开源向量库，性能优秀，部署简单。

**核心特性**：
- 单二进制部署，无需外部依赖
- Payload（元数据）原生支持
- 丰富的过滤语法（must、should、range）
- 内置量化（Scalar Quantization）降低内存

**适用场景**：
- 中小规模（千万级向量）
- 追求部署简单
- 对内存敏感

### 2.4 Weaviate

GraphQL 接口的开源向量库，集成度高。

**核心特性**：
- 内置 embedding 模型（也可外挂）
- GraphQL API
- 混合检索（向量 + BM25）
- 模块化架构（多种向量化模块）

**适用场景**：
- 希望开箱即用（不自己管 embedding）
- 全栈 JS / TypeScript 团队

### 2.5 Elasticsearch

老牌搜索引擎，从 8.0 开始原生支持向量检索。

**核心特性**：
- 混合检索（BM25 + 向量）原生支持
- Lucene 引擎稳定成熟
- 大规模集群运维经验丰富
- 强大的聚合分析能力

**适用场景**：
- 已有 ES 集群
- 需要复杂的全文搜索 + 向量混合
- 大团队、高 SLA 场景

## 三、关键性能指标对比

| 方案 | 100 万向量查询延迟 | 1000 万向量内存占用 | 部署复杂度 |
|---|---|---|---|
| pgvector | 5-20ms | 4-8 GB | 低（PG 扩展） |
| Milvus | 2-10ms | 8-16 GB | 高（多组件） |
| Qdrant | 3-15ms | 2-6 GB（含量化） | 低（单二进制） |
| Weaviate | 5-20ms | 6-12 GB | 中 |
| Elasticsearch | 10-30ms | 8-16 GB | 中高 |

> 数据基于 1024 维向量、HNSW 索引、ef_construction=64 的典型配置。

## 四、选型决策树

```
团队规模？
├─ 个人 / 小团队（< 5 人）
│   └─ 数据量？
│       ├─ < 100 万 → pgvector（最简单）
│       └─ > 100 万 → Qdrant（单机扛得住）
├─ 中型团队（5-30 人）
│   └─ 是否已有 PostgreSQL？
│       ├─ 是 → pgvector（先跑起来，不够再换）
│       └─ 否 → Qdrant 或 Weaviate
└─ 大型团队（30+ 人）
    └─ 数据量？
        ├─ 亿级 → Milvus（分布式必需）
        └─ 千万级 → Elasticsearch（与现有日志栈统一）
```

## 五、HNSW 索引参数调优

HNSW（Hierarchical Navigable Small World）是几乎所有向量库都支持的索引算法，核心参数：

### 5.1 m（连接数）

每个节点在图中的连接数。越大召回越高、内存占用越大。推荐 16-32。

### 5.2 ef_construction（构建时搜索深度）

建索引时的搜索深度。越大构建越慢、质量越好。推荐 64-200。

### 5.3 ef_search（查询时搜索深度）

查询时的搜索深度。越大召回越高、延迟越长。推荐 40-100。

经验调优路径：
1. 固定 m=16, ef_construction=64
2. 测试不同 ef_search（40 → 80 → 120）的 Recall@10 和延迟曲线
3. 选择满足延迟 SLA 的最大 ef_search

## 六、混合检索的实现

纯向量检索的短板是"字面匹配"能力弱（如查具体产品型号）。混合检索结合 BM25 和向量，互相补足。

### 6.1 PostgreSQL 中的混合检索

```sql
-- tsvector 全文索引 + pgvector 余弦索引
SELECT id, content,
       ts_rank_cd(fts, query) AS bm25_score,
       1 - (embedding <=> '[...]') AS vec_score
FROM knowledge_chunk
WHERE fts @@ query OR 1 - (embedding <=> '[...]') > 0.7
ORDER BY (bm25_score / (bm25_score + 1)) + vec_score DESC
LIMIT 20;
```

### 6.2 RRF 融合方式

更鲁棒的做法是两路独立检索 topK，再用 RRF 融合排名（详见《检索重排流水线设计》一文）。

## 七、迁移成本评估

向量数据库一旦上线，迁移成本极高：
- 数据需重新生成（embedding 模型可能变了）
- API 不兼容，业务代码全改
- 运维工具（监控、备份）需重建

建议：
1. 选型时把"未来 3 年数据量"估到 3 倍
2. 优先选 SQL 接口的方案（pgvector）便于迁移
3. 业务层抽象检索接口（如本项目 `RetrievalService`）

## 八、总结

向量数据库选型不是越新越好、越快越好。综合考虑：
- 数据规模增长预期
- 团队运维能力
- 与现有基础设施的协同
- 迁移成本

对于大多数企业内部 RAG 系统，**pgvector 是性价比最高的起点**——简单、稳定、与业务库共用，数据量到千万级之前都不需要换。
