# 检索重排流水线设计

> 现代 RAG 检索已不再是"一次向量召回就完事"，而是一条多阶段流水线：双路召回 → 融合 → 精排 → 上下文增强。本文系统梳理每个阶段的作用、实现和工程权衡。

## 一、为什么不直接用向量检索

纯向量检索（Bi-Encoder）虽然快，但有明显短板：

1. **语义近似 ≠ 问题相关**：向量接近只表示"表述相似"，不代表回答了用户问题
2. **关键词盲区**：对专有名词、术语、人名等"字面命中"型查询效果差
3. **长尾召回低**：训练数据中少见的概念，embedding 质量不稳定

所以实际系统会在向量召回之后，叠加 BM25 关键词召回，再用 Cross-Encoder 精排。

## 二、双路召回

### 2.1 向量路（Bi-Encoder）

用 embedding 模型把 query 和 chunk 都转成向量，算余弦相似度。

- **优点**：chunk 向量可离线预算，查询时只需算 query 向量，延迟低（< 50ms）
- **缺点**：query 和 chunk 独立编码，无法捕捉两者间的细粒度交互
- **典型 topK**：20（候选池给后续 rerank 用）

### 2.2 关键词路（BM25）

基于 TF-IDF 改进的经典算法，用 PostgreSQL 的 `tsvector + plainto_tsquery` 或 Elasticsearch 实现。

- **优点**：对专有名词、人名、产品型号等精确匹配强；零学习成本
- **缺点**：无法理解同义词（查"裁员"找不到"优化"）
- **中文分词**：PostgreSQL 配合 zhparser 扩展；Elasticsearch 配合 IK 分词器

### 2.3 查询扩展：弥补 BM25 的同义词盲区

在 BM25 检索前，先基于 embedding 从词汇表中找 query 的 top-K 同义词，拼成 `"原query OR 同义词1 OR 同义词2"` 喂给 `websearch_to_tsquery`。这样查"裁员"也能召回含"裁员优化"、"裁员赔偿"的文档。

### 2.4 多查询扩展：覆盖不同视角

调 LLM 把原 query 改写为 3 个不同视角的子 query（不是同义词替换），每个子 query 都做双路召回，结果统一融合。如：

- 原始："怎么实现 RAG 切块策略"
- 改写 1："RAG chunking 有哪些常见方案"
- 改写 2："知识库切分文档的最佳实践"
- 改写 3："检索增强生成系统如何分块处理文本"

## 三、RRF 融合

Reciprocal Rank Fusion（倒数排名融合）是双路结果合并的经典算法：

```
score(d) = Σ_path 1/(k + rank_path(d))
```

其中 `k` 是平滑常数（通常取 60），`rank_path(d)` 是文档 d 在某一路召回结果中的排名（从 1 开始）。

### 3.1 为什么用 RRF 而不是分数加权

- 向量相似度是 [0,1] 的浮点，BM25 是非归一化的相关度分，两者不可直接比
- RRF 只用排名信息，对分数分布不敏感，鲁棒性强
- 计算简单，无需训练融合参数

### 3.2 多查询场景下的 RRF 改造

每个子 query 的双路结果都独立计 rank，跨 query 同 chunk 的分数累加：

```
score(d) = Σ_query Σ_path 1/(k + rank_{query,path}(d))
```

这样避免了"拼接大 list 导致子 query N 的 chunk rank 偏低"的不公平。

## 四、Cross-Encoder 精排

### 4.1 Bi-Encoder vs Cross-Encoder

| 维度 | Bi-Encoder | Cross-Encoder |
|---|---|---|
| 编码方式 | query 和 chunk 独立编码 | query 和 chunk 拼一起送入模型 |
| 延迟 | 极快（chunk 向量离线预算） | 慢（每对都要前向） |
| 精度 | 中 | 高 |
| 用途 | 大规模召回 | 小规模精排 |

### 4.2 典型 rerank 模型

- **gte-rerank-v2**（阿里 DashScope）：中文友好、API 即服务
- **bge-reranker-large**：开源可自部署
- **cohere/rerank-english-v3.0**：英文场景标杆

### 4.3 候选池大小

太大稀释 reranker 注意力，太小错过相关结果。经验值：20 条。

## 五、父子节点注入

rerank 后，对前 topN 个高分父节点（章节树场景），按 `parent_id` 反查直接子节点（最多 2 个），子节点分数 = 父 × 0.5 折扣。

### 5.1 为什么需要注入

章节树切片下，命中 H1 时其 H2 子章节往往也相关但可能未被独立召回。注入保证上下文完整性。

### 5.2 分数折扣的考量

折扣 0.5 是经验值，目的：
- 父必然排在自己子节点前
- 但子节点可能挤掉其他低分父节点（设计如此，扩展语义的代价）

## 六、Token 预算控制

rerank + 注入后的 chunks 不能全塞给 LLM，否则会爆上下文窗口或触发 "Lost in the middle" 注意力衰减。

### 6.1 四级降级策略

按节点优先级顺序消费预算（高分节点先占预算）：

1. 完整 content ≤ 预算 → 用 content
2. content 超 但 summary ≤ 预算 → `[摘要]` + summary
3. 都超 但预算 ≥ 头+尾+分隔 → 头 500 + `[省略中段]` + 尾 500
4. 极小预算 → `content.substring(0, budget-suffix)` + `...`

### 6.2 默认预算

3000 字符（保守值，RAG 不应占满 LLM 窗口）。中文约等于 4500 token，英文约 750 token。

## 七、整体流水线示例

| 阶段 | 输入 | 输出 | 延迟 |
|---|---|---|---|
| 多查询扩展 | 1 个原 query | 4 个子 query（含原） | ~1.5s |
| 双路召回 | 4 个子 query | 4×(20 向量 + 20 BM25) = 160 候选 | ~200ms |
| RRF 融合 | 160 候选 | 20 候选池 | <10ms |
| Cross-Encoder | 20 候选 + 原 query | 20 候选（按精排分重排） | ~500ms |
| 父子注入 | 20 候选 | 20-30 候选 | <50ms |
| Token 预算 | 20-30 候选 | ≤ 3000 字符的 context | <10ms |

端到端延迟 ~2-3 秒，可接受。

## 八、总结

检索流水线是 RAG 系统的精度上限——下游 LLM 再强，检索不到相关内容也是巧妇难为无米之炊。每个阶段都有可选优化空间（缓存、并发、量化），但核心算法已经稳定。后续优化的重点是工程化：延迟降低、成本控制、可观测性。
