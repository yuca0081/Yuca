# RAG 系统架构总览

> 检索增强生成（Retrieval-Augmented Generation, 简称 RAG）是当前大语言模型落地企业知识库的主流方案。本文梳理一个生产级 RAG 系统的核心组件、数据流和常见工程权衡。

## 一、为什么需要 RAG

大语言模型（LLM）虽然有很强的语言理解和生成能力，但在企业场景中存在三类先天缺陷：

1. **知识截止**：模型训练数据有时间边界，无法回答最新信息
2. **领域盲区**：企业内部文档、业务规则、产品手册等私有知识模型从未见过
3. **幻觉风险**：模型对不熟悉的问题会"一本正经地胡说八道"

RAG 的核心思路是：在生成回答前，先从外部知识库检索相关片段，把检索到的内容作为上下文塞给 LLM，让模型基于"已知事实"回答。这样既保留 LLM 的语言能力，又解决了上述三个问题。

## 二、核心组件

一个完整的 RAG 链路通常包含五个核心组件：

### 2.1 文档处理模块

负责把原始文档（PDF、Word、Markdown、HTML 等）解析为纯文本，再切成适合做 embedding 的语义单元。切块策略直接影响检索质量——太大稀释信号，太小丢失上下文。

### 2.2 嵌入服务

把文本转成稠密向量。常用模型包括 OpenAI 的 text-embedding-3、DashScope 的 text-embedding-v3、BGE 系列等。维度通常在 768 到 1536 之间。

### 2.3 向量数据库

存储文档向量并支持快速相似度检索。常见选型：
- PostgreSQL + pgvector（轻量、与业务库共用）
- Milvus / Weaviate / Qdrant（专用向量库，性能更强）
- Elasticsearch（关键词+向量混合）

### 2.4 检索流水线

现代 RAG 已不只是单次向量召回，而是多阶段流水线：
- 双路召回（向量 + BM25 关键词）
- RRF 融合
- Cross-Encoder 精排
- 父子节点注入

### 2.5 生成模块

把检索结果和用户问题拼成 prompt，交给 LLM 生成最终回答。包含 Token 预算控制（防止上下文溢出）和 Lost-in-the-middle 缓解策略。

## 三、典型数据流

```
用户提问
  ↓
[查询扩展]（同义词 / 多查询改写）
  ↓
[双路召回] 向量检索 + 关键词检索
  ↓
[RRF 融合]
  ↓
[Cross-Encoder 精排]
  ↓
[父子节点注入] 补全章节上下文
  ↓
[Token 预算控制]
  ↓
[LLM 生成]
  ↓
用户看到的回答
```

## 四、常见工程权衡

### 4.1 切块大小 vs 检索精度

- **小切片（100-300 字符）**：信号集中，向量精度高，但可能丢失上下文
- **大切片（800-1500 字符）**：上下文完整，但向量信号被稀释
- **推荐策略**：章节树独占切片 + LLM 摘要做 embedding + 父子注入补全上下文

### 4.2 召回率 vs 准确率

- **召回率优先**：放宽阈值、增加 topK、多路召回，适合"宁可错杀不可放过"
- **准确率优先**：严格阈值、Cross-Encoder 精排，适合"宁可少给不能错"

### 4.3 实时性 vs 成本

每次检索都调 LLM 做查询扩展会带来 1-2 秒延迟。可以加 Redis 缓存近期查询扩展结果，命中率高的场景能省掉 80% 的 LLM 调用。

## 五、常见评估指标

| 指标 | 含义 | 典型值 |
|---|---|---|
| Recall@5 | 前 5 条结果中命中相关文档的比例 | > 0.85 |
| MRR | 第一个相关结果的平均倒数排名 | > 0.65 |
| 端到端延迟 | 用户提问到回答返回 | < 3 秒 |
| 上下文命中率 | 检索片段中含答案的比例 | > 0.9 |

## 六、总结

RAG 不是单一技术，而是一套工程化体系。每个环节都有多种选择，最终效果取决于各环节的协同调优。后续文档会分别深入每个组件的实现细节。
