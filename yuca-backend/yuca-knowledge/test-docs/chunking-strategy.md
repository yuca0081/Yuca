# 文档切块策略深度解析

> 切块（Chunking）是 RAG 系统中最容易被忽视、但影响最显著的工程环节。同样的文档、同样的 embedding 模型，不同的切块策略能让检索准确率波动 20% 以上。本文系统梳理常见切块方案及其适用场景。

## 一、切块的本质矛盾

切块要解决的核心矛盾是：**embedding 模型的信号集中性 vs 检索结果的上下文完整性**。

- embedding 模型对短文本（200-500 字符）的语义压缩效果最好，长文本会让向量信号被稀释
- 但用户问问题时期望得到有完整上下文的答案，孤立的短片段往往无法独立解释

如何在这两端找到平衡？答案是**独占式切片 + 父子关系 + 摘要 embedding**。

## 二、主流切块方案对比

### 2.1 固定长度切片

最朴素的方案：按固定字符数切片，相邻片段保留一定重叠。

**优点**：实现简单、切片数稳定、不依赖文档结构

**缺点**：完全无视语义边界，可能在句子中间切断，产生无意义片段

**适用场景**：纯文本日志、无结构的 TXT 文件、降级兜底

### 2.2 段落切片

按段落（双换行）分段，超长段落再二次切。

**优点**：保留段落语义、对 Markdown / HTML 等结构化文档友好

**缺点**：段落长度参差不齐，过短段落会产生弱信号 embedding

**适用场景**：技术文档、博客文章、产品说明

### 2.3 章节树切片

针对 Markdown 这类有 H1-H6 标题层级的文档，按标题层级构建树形结构，每个节点只存"自己标题后到下一标题前的正文"（独占式切片），不含子节点正文。

**优点**：
- 父子关系天然清晰（H1 下有 H2，H2 下有 H3）
- 每个节点有独立标题和面包屑路径，向量信号强
- 命中父节点时可运行时反查子节点补全上下文

**缺点**：依赖文档规范使用标题，乱用 H1-H6 的文档切片质量差

**适用场景**：技术规范、API 文档、Wiki、产品手册

## 三、章节树切片的实现细节

### 3.1 独占式切片的存储模型

每个节点存储：
- `title`：章节标题（如"切块策略"）
- `heading_level`：层级（1-6 对应 H1-H6）
- `breadcrumb`：面包屑路径（如"RAG > 召回 > 切块策略"）
- `content`：独占正文（自己标题后到下一任意标题前的内容）
- `parent_id`：父节点 DB id
- `line_start` / `line_end`：源文件行号，便于精确溯源

### 3.2 LLM 摘要做 embedding

短节点（如纯目录章节）的 content 很短，直接做 embedding 信号弱；长节点 content 又会让向量稀释。

解决方案：用便宜的小模型（如 qwen-turbo）为每个章节节点生成 ≤200 字的摘要，用 summary 字段做 embedding，content 仍保留独占原文用于检索展示。

### 3.3 父子节点注入

rerank 之后，对前 topN 个高分父节点，按 `parent_id` 反查直接子节点（最多 2 个），子节点分数 = 父节点分数 × 0.5 折扣。这样命中的 H1 章节会自动带上其 H2 子章节，扩展语义。

## 四、切块参数调优经验

### 4.1 切片长度

- 章节树模式：自然长度（不强求统一），依赖摘要 embedding 而非原文
- 段落模式：单段 ≤ 500 字符，超出二次切
- 字符模式：默认 100 字符 + 10 重叠，适合降级兜底

### 4.2 重叠量

固定长度切片的重叠量推荐 10-15%。重叠太少会切断语义，太多会增加存储和 embedding 成本。

### 4.3 是否启用摘要

| 文档类型 | 是否启用摘要 | 原因 |
|---|---|---|
| 技术文档（章节清晰） | 启用 | 短章节信号弱，摘要能放大语义 |
| 代码文档（含大段代码） | 不启用 | 代码摘要意义不大，反而引入噪声 |
| 对话日志 | 不启用 | 对话本身就是离散语义单元 |

## 五、常见踩坑

### 5.1 把目录页当章节

有些 Markdown 文档的"目录"是 `- [xxx](#xxx)` 形式的链接列表，被章节树构造器识别为 H1/H2 的子节点后，会产生大量无内容的占位节点。解决：上传前用脚本清洗目录章节。

### 5.2 标题层级跳跃

`# 一级标题` 直接跟 `### 三级标题`（跳过 H2），章节树会构造出"无父的孤儿节点"。解决：在构造器里补一个虚拟 H2 容器，或者上传前规范文档。

### 5.3 PDF 表格被拉平

PDF 解析后表格通常被拉平成"单元格1 | 单元格2 | 单元格3"的纯文本，失去行列语义。解决：用专门的 PDF 表格解析器（如 Camelot、Tabula）预处理。

## 六、总结

切块没有银弹。生产系统的做法是：
1. 对每类文档评估其结构化程度
2. 选用对应的切块策略（章节树 / 段落 / 字符）
3. 配合质量评分自动路由（Clean 走章节树，Garbage 走字符切片兜底）
4. 持续用评估集回归测试
