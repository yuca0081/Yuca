## P0 - 基础能力缺失（影响生产可用性）

### 1. PDF / DOCX 文件解析修复 ✅ 已完成（2026-07-20）

**原问题**：`pom.xml` 已引入 `poi-ooxml` 和 `pdfbox`，但 `KnowledgeDocService.uploadDocument` 直接把文件字节当 UTF-8 字符串解析：

```java
// KnowledgeDocService.java:100
byte[] bytes = file.getBytes();
String content = new String(bytes, StandardCharsets.UTF_8);
```

PDF / DOCX 是二进制格式，按 UTF-8 解码会得到乱码，章节树构建会失败或产生无意义切片。

**实施结果**：

新增 4 个解析器文件（位于 `document/parser/`）：
- `DocumentParser.java` — `@FunctionalInterface`，定义 `String parse(byte[] bytes)`
- `MarkdownTxtParser.java` — 处理 md / txt，等价原 UTF-8 解码行为
- `PdfParser.java` — 基于 PDFBox 2.0.29，`PDDocument.load` + `PDFTextStripper`，加密 PDF 直接抛 `IOException`
- `DocxParser.java` — 基于 POI 5.2.5，按 `IBodyElement` 顺序遍历段落和表格，表格转纯文本时单元格用 ` | ` 分隔
- `DocumentParserRegistry.java` — 按扩展名路由，未知扩展名降级到 MarkdownTxtParser 并记 WARN

修改的文件：
- `config/KnowledgeConfig.java` — 新增 `documentParserRegistry` Bean，注册 md/txt/pdf/docx 四个扩展名
- `service/KnowledgeDocService.java` — 注入 `DocumentParserRegistry`，把 `new String(bytes, UTF_8)` 替换为 `registry.parse(fileFormat, bytes)`；顺手修复原 `catch (IOException) → throw RuntimeException` 的异常吞掉问题，改为统一 `BusinessException`

**范围说明**：
- 不含 OCR 兜底（扫描件 PDF 仍返回空文本，留给 P2-#12）
- 不含 `.doc` 老版二进制格式（需 HWPFDocument）
- 表格在 DOCX 中以纯文本形式拼接，无 H1-H6 标题，自然走字符切片降级路径

**验证状态**：因 `yuca-ai/SummaryEnhancer.java` 在研代码编译错误阻塞，**未跑 mvn compile**。需待该文件修复后补一次全量编译验证。

### 2. 文档增量更新（SHA256 指纹对比） ✅ 已完成（2026-07-21）

**原问题**：每次上传都走"新建"流程，无去重和更新逻辑。用户重新上传同名文档会让旧切片和新切片并存，检索结果被旧版本污染；误重复上传则浪费 embedding 成本。

**实施结果**：

新增字段：
- DDL `yuca-app/src/main/resources/table/knowledge/knowledge_doc.sql` 增加 `content_hash VARCHAR(64)` + `idx_doc_hash` 普通索引 + 表/列注释
- 实体 `entity/KnowledgeDoc.java` 增加 `private String contentHash;`

三分支路由（`KnowledgeDocService.uploadDocument`）：
- **分支 A（hash 完全相同）**：跳过解析/上传/嵌入，直接返回旧 docId。日志 `[upload] hash 相同，跳过上传`
- **分支 B（首次上传）**：走原新建流程。日志 `[upload] 新建文档`
- **分支 C（hash 不同）**：触发重建——同一 `@Transactional` 内软删旧 doc + 软删旧切片 + 上传 MinIO（同路径覆盖）+ 插入新 doc（带 hash）+ 插入新切片。日志 `[upload] 检测到内容变更，触发重建`

新增 3 个私有方法：
- `sha256Hex(byte[])` — 基于 `HexFormat`（Java 17+），比 `String.format("%02x")` 快一个数量级
- `findActiveByName(kbId, docName)` — 用 LambdaQueryWrapper + `selectOne`，依赖 `@TableLogic` 自动过滤 deleted=1
- `softDeleteChunksOfDoc(docId)` — 用 `knowledgeChunkMapper.delete(wrapper)`，`@TableLogic` 自动转 `UPDATE SET deleted=1`

**关键决策**：
- 识别口径用 `(kbId, docName)` 而非全局 hash，符合用户视角
- 顺序优化：先算 hash → 查旧 → 决定是否解析，hash 相同时完全跳过解析成本
- MinIO 文件覆盖（同路径上传），不保留备份——节省存储，旧 DB 记录仍支持逻辑审计
- 接口零变更：返回值仍为 `Long` docId，前端无感

**范围说明**：
- 不含显式 `reupload` 接口（未来扩展）
- 不含 MinIO 旧文件清理逻辑（留给 P0-#3）
- 不含状态机/异步重建（留给 P2-#13）
- 不含 `(kbId, docName)` 数据库唯一约束（并发上传仍可能产生重复）

**遗留风险**：
- `findActiveByName` 用 `selectOne`，若历史数据已存在多条未删除同名记录会抛 `TooManyResultsException`。这是有意为之——让数据污染显式暴露而非默默选第一条
- 重建同步执行，大文档耗时长（嵌入 30s+），异步化留给 P2-#13
- 进程崩溃在重建中途：MinIO 已是新文件但 DB 可能保留旧记录的 deleted=0，下次上传会再次"重建"——可接受

### 3. 文档删除时清理 MinIO 原文件 ✅ 已完成（2026-07-21）

**原问题**：`KnowledgeDocService.deleteDocuments` 只做 DB 软删，MinIO 物理文件永久残留，造成存储成本累积。

**实施结果**：

修改 `service/KnowledgeDocService.java`：

- `deleteDocuments` 在 DB 软删 + 切片软删之后，调用 `deleteMinioFileSafely(doc.getFilePath(), docId)`
- 新增私有方法 `deleteMinioFileSafely`：
  - 判空（`file_path` 为 NULL 或 blank 时跳过 + DEBUG 日志）
  - try-catch 包裹 `fileStorageService.delete()`（任何异常仅记 WARN 日志，不抛出）
  - 失败残留文件留作 P2-#13 状态机时的后台清理 Job 兜底

**关键决策**：
- 顺序：DB 软删 → MinIO 删。MinIO 失败时 DB 已成功，业务功能正常
- 失败隔离：MinIO 异常不影响事务，最终一致即可
- 复用 `FileStorageService.delete`（接口已存在，`MinioFileStorageService` 的 `removeObject` 幂等）
- 不动 Controller / Mapper / DDL

**范围说明**：
- 重建场景的 MinIO 清理不处理（P0-#2 已用同路径覆盖语义）
- 不引入 `@TransactionalEventListener` 事务后置（保持简单）
- 不引入异步孤儿清理 Job（留给 P2-#13）

**遗留风险**：
- DB 事务提交前 MinIO 已删，若事务回滚会丢失文件——概率极低，且 P0-#2 增量更新可恢复
- for 循环部分成功部分失败——当前实现已有行为，未改变

### 5. content 字段保留完整原文 ⊘ 取消（2026-07-21，已被 P1-#4 覆盖）

**取消原因**：重新精读简历原意后修正解读。

简历原文：
> 父节点只存**引言**（到下一级标题为止）**不含子节点正文**，避免父子 embedding 信号重复……针对独占式切片隐患，每节点用 LLM 生成摘要做 embedding、content 字段保留**完整原文**

"完整原文"的对比对象是"摘要"而非"独占切片"。即：
- `summary`（LLM 摘要）→ embedding
- `content`（独占切片原文，非摘要）→ 检索展示

这正是 P1-#4 已实现的状态：embedding 用 summary、content 保持独占切片原文不变。本项**已被 P1-#4 覆盖**。

**原方案的错误**：误读为"content 含所有子孙的完整正文"，会导致父节点 content 嵌套包含子节点 content，存储冗余约 185%（粗算 3400 字原文 → 9700 字存储）。违背简历"父节点不含子节点正文"原则。

**命中父节点时如何拿到完整上下文**：交给 P1-#6 保守父子注入——检索阶段按 `parent_id` 查子孙 chunk 运行时拼装，零存储冗余。

### 6. 保守父子注入 ✅ 已完成（2026-07-21）

**原问题**：rerank 后直接截取 topN 返回。章节树父子关系完全未被利用——命中 H1 时 H1 的子节点（H2/H3）即使相关也不会自动出现。

**实施结果**：

修改 3 个文件：

1. **`mapper/KnowledgeChunkMapper.java`** — 新增 `selectChildrenByParentId(parentId, limit)` 方法声明

2. **`resources/mapper/knowledge/KnowledgeChunkMapper.xml`**
   - 新增 selectChildrenByParentId SQL：按 chunk_index ASC，LIMIT #{limit}，复用 idx_chunk_parent 索引
   - 顺手补全 BaseResultMap 的章节树字段映射（title/heading_level/breadcrumb/parent_id/summary/line_start/line_end）——之前依赖 autoMapping 不显式

3. **`service/KnowledgeRetrievalService.java`**
   - 新增常量 `MAX_CHILDREN_PER_PARENT=2`、`CHILD_SCORE_DISCOUNT=0.5`
   - `reciprocalRankFusion` 增加 `parentIdMap` 出参；新增 `collectParentId` 辅助方法（parentId 非 null 时才收集）
   - `retrieve` 步骤 5 调 `injectChildren`，步骤 6 截 topN（流程从 5 步变 6 步）
   - 新增 `injectChildren` 方法：去重 → 循环前 topN 个父节点查子节点 → 合并按 score 重排
   - 类 Javadoc 从"三级流水线"更新为"四级流水线"

**关键决策**：
- 触发时机：rerank 后、limit 前（rerank 分数已写入 score，子节点分数 = 父 × 0.5 语义稳定）
- 触发范围：仅前 topN 个高分父节点（避免低分节点无意义扩展）
- 每父注入：最多 2 个直接子节点（按 chunk_index ASC）
- 分数折扣：0.5（保证父必然排在自己子节点前，但可能挤掉其他低分节点）
- 去重：existingIds 防止子节点被重复召回
- 扁平切片 / 根节点：parentIdMap 不含该 chunkId → 跳过注入

**范围说明**：
- 不含孙节点注入（仅直接子节点）
- 不含批量查 SQL 优化（topN 默认 5，5 次索引查询延迟可忽略）
- 不含折扣系数 / 子节点数配置化
- 不含检索结果展示端区分父/子节点（前端看到的都是 RetrievedChunk）

**遗留风险**：
- rerank 失败降级时 score 是 RRF 分数（很小，~1/60 级别），子节点更小但仍合理
- 子节点可能挤掉其他低分父节点——设计如此，扩展语义的代价

### 7. Token 预算控制 ✅ 已完成（2026-07-21）

**原问题**：`RagEnhancer.buildRagContext` 直接把所有 chunks 拼进去，长文档命中可能爆 LLM 上下文或触发 "Lost in the middle" 注意力衰减。

**实施结果**：

修改 4 个文件 + 新建 1 个：

1. **`yuca-ai/.../retrieval/RetrievedChunk.java`**（修改）
   - 加 `summary` 字段
   - 5 参构造器委派 6 参（传 null），向后兼容
   - 6 参构造器用于 KnowledgeRetrievalService 透传 summary

2. **`yuca-ai/.../retrieval/TokenBudgetAssembler.java`**（**新建**）
   - `@Component`，默认 3000 字符预算
   - 四级降级策略：
     1. 完整 content ≤ 预算 → 用 content
     2. content 超但 summary ≤ 预算 → `[摘要]` + summary
     3. 都超但预算 ≥ 头+尾+分隔 → 头 500 + `[省略中段]` + 尾 500
     4. 极小预算 → `content.substring(0, budget-suffix)` + `...`
   - 顺序消费预算，耗尽即停止（高分节点优先占预算）

3. **`yuca-knowledge/.../service/KnowledgeRetrievalService.java`**（修改）
   - `reciprocalRankFusion` 的 `new RetrievedChunk(...)` 补 summary 参数
   - `injectChildren` 的 `new RetrievedChunk(...)` 补 summary 参数

4. **`yuca-ai/.../agent/enhancer/RagEnhancer.java`**（修改）
   - 构造器加 `TokenBudgetAssembler` 参数
   - 删除 `buildRagContext` 私有方法
   - `assemble(chunks)` 替代直接拼接

5. **`yuca-ai/.../agent/AgentFactory.java`**（修改）
   - 注入 `tokenBudgetAssembler` 字段（@RequiredArgsConstructor 自动）
   - `defaultAgent` 创建 RagEnhancer 时传入 assembler

**关键决策**：
- Tokenizer：暂不引入 jtokkit（Qwen 词表不匹配，估算有偏差），用字符数估算（中文 1 字 ≈ 1.5 token）。注释里说明语义等价 ~2000-4500 token 视中英文比
- 默认预算：3000 字符（保守值，RAG 不应占满 LLM 窗口）
- 头尾保留：各 500 字符（开头痛点 + 结尾结论）
- Assembler 放 yuca-ai：与 RetrievedChunk 同包，便于其他模块复用
- RagEnhancer 不再是 @Component 自注入：通过 AgentFactory 注入 assembler 后传给 RagEnhancer 构造器

**范围说明**：
- 不引入精确 tokenizer（jtokkit）
- 预算值未配置化（先硬编码 3000）
- 不做不同场景的差异化预算（QA vs CREATION）
- AiTestController 不改（仅测试用）

**遗留风险**：
- 字符估算对中英文混合偏差：3000 字符 ≈ 4500 token（纯中文）或 750 token（纯英文），偏差可接受
- 预算耗尽时低分 chunks 丢失：设计如此，按 score 排序后顺序消费

### 8. 查询扩展（同义词自动发现） ✅ 已完成（2026-07-22）

**原问题**：BM25 路用原始 query 做 `plainto_tsquery`，无法覆盖同义词（如查"裁员"找不到"裁员优化"）。

**实施结果**：

新建 5 个文件 + 修改 4 个文件：

1. **`yuca-app/.../table/knowledge/knowledge_vocabulary.sql`**（新建）
   - 表 `knowledge_vocabulary`：`id / kb_id / doc_id / term / embedding(vector1024) / source / created_at / updated_at / deleted`
   - `doc_id` 可空：NULL=管理员预设（SQL 手动塞）；非 NULL=自动抽取（便于级联清理）
   - HNSW 向量索引（与 knowledge_chunk 同构，部分谓词 `WHERE deleted=0`）+ kb/doc/deleted 普通索引

2. **`entity/KnowledgeVocabulary.java`**（新建）
   - 标准实体：`@TableName / @TableId(AUTO) / @TableLogic`
   - `embedding` 用 `PGVectorTypeHandler`（与 KnowledgeChunk 一致）

3. **`mapper/KnowledgeVocabularyMapper.java` + `mapper/knowledge/KnowledgeVocabularyMapper.xml`**（新建）
   - 唯一方法 `searchSimilar(kbId, queryEmbedding, topK, threshold)`
   - SQL：按 `deleted=FALSE AND kb_id=?` 过滤，余弦距离 HNSW 召回

4. **`service/QueryExpansionService.java`**（新建）
   - 入口：`expand(query, kbId)` → 返回 "原query OR 同义词1 OR 同义词2" 形态的串

### 9. 多查询扩展（Multi-Query Expansion） ✅ 已完成（2026-07-22）

**原问题**：单 query 召回视角单一，LLM 改写可覆盖不同表达和术语视角。

**实施结果**：

新建 1 个文件 + 修改 2 个文件：

1. **`service/MultiQueryExpander.java`**（新建）
   - 入口：`List<String> rewrite(String query)` 返回 N 个**不同视角**改写（**不含原 query**，调用方手动拼到第一位便于复用 embedding）
   - 常量：`REWRITE_COUNT=3`（简历下限）、`MAX_REWRITE_LEN=200`（超长视为 LLM 跑偏过滤）
   - `@Value("${yuca.knowledge.multi-query.enabled:true}")` 开关，false 时返回空列表
   - prompt 硬编码：text block 模板，强调"不同视角、保持原意、每行一个、不加序号"
   - LLM 模型：复用 `aiProperties.getSummary().getModelName()`（qwen-turbo，便宜快），通过 `agentFactory.simpleAgent(modelName)` 调用
   - 输出解析：split `\n` → trim → strip 序号前缀（`1.` / `1、` / `1)` / `- ` / `* `）→ 长度过滤 [2,200] → 与原 query 等值过滤 → LinkedHashSet 去重 → limit 3
   - 失败降级：catch + WARN，返回 `Collections.emptyList()`

2. **`service/KnowledgeRetrievalService.java`**（修改）
   - 注入 `MultiQueryExpander`
   - `retrieve` 改造（流程从 6 步增到 6 步，但召回逻辑重写）：
     - 步骤 1：组装 queries 列表，原 query 永远第一位，追加 `multiQueryExpander.rewrite(query)`，再做一次等值去重
     - 步骤 2：循环每个子 query 双路召回——向量路用子 query 本身 embed（语义捕获足够），BM25 路过 #8 扩展拼 OR 串（最大化召回覆盖）
     - 步骤 3：调用新方法 `reciprocalRankFusionMultiQuery` 做跨 query 累加 RRF
     - 步骤 4：rerank 仍用**原始 query**（不是子 query；rerank 关注"对原始意图的相关性"）
     - 步骤 5-6：父子注入 / topN 截取不变
   - 删除旧 `reciprocalRankFusion(vectorResults, keywordResults, ...)`，新增 `reciprocalRankFusionMultiQuery(vectorResultsPerQuery, keywordResultsPerQuery, ...)`：
     - 入参从两个扁平 list 改为两个 list-of-list（每子 query 一组）
     - 公式：`score(d) = Σ_{q} Σ_{path} 1/(k + rank_{q,path}(d))`，每子 query 每路都从 rank=1 起算
     - 避免"拼接大 list 导致子 query N 的 chunk rank 偏低"的不公平
   - 类 Javadoc 更新说明多查询 × 双路召回

3. **`application.yml`**（修改）
   - 新增 `yuca.knowledge.multi-query.enabled=true`（与 `query-expansion.enabled` 同级，两开关独立）

**关键决策**：
- 改写数量取 3：简历下限，避免延迟（每子 query +1 次 embed + 2 次 DB 查询）
- 保留原 query 作为第一路：兜底精确匹配，防 LLM 改写跑偏
- 与 #8 叠加：每个子 query 的 BM25 路都过 #8，最大化召回覆盖
- 跨 query RRF 累加：每子 query 独立计 rank，多 query 同时命中的 chunk 分数自然变高（语义聚焦）
- rerank 用原 query：精排关注"对原始意图的相关性"，子 query 只负责召回广度
- 串行召回：N=3 时 3+3=6 次 DB 查询 + 3 次 embed + 1 次 LLM，延迟 ~2-3s 可接受；并行优化留后续

**范围说明**：
- 不含改写结果缓存（每次实时调 LLM，高频场景后续可加）
- 不含并发召回（串行，CompletableFuture 优化留后续）
- 不含改写数量 / prompt 配置化（先硬编码）
- 不含改写质量评分（不丢弃"跑偏"的改写，依赖 RRF 自然衰减）

**遗留风险**：
- LLM 延迟：qwen-turbo 单次 ~1-2 秒，多查询扩展让检索首字节 +1-2s（RAG 非流式可接受）
- 改写跑偏：保留原 query 兜底；多 query 同时命中相关 chunk 时 RRF 自然加权
- 成本累积：每次检索多 1 次 LLM + 3 次 embed 调用；DashScope 价格低，高频场景可加缓存
- 子 query 重复：双 `contains` 去重（rewrite 内部 + retrieve 外部各一次），覆盖 LLM 输出与原 query 等值的情况

### 15. 章节树空 content 节点过滤 ✅ 已完成（2026-07-23）

**原问题**：`MarkdownChapterTreeBuilder` 的"独占式切片"遇到**连续标题**（H1 紧接 H2、H2 紧接 H3，中间无正文）时，父节点 `content` 被赋为空字符串。这些空 content 节点仍然会进 `flattenDfs` → 生成 chunk → 调 embedding → 入库，导致：
- 检索命中该 chunk 时 LLM 上下文为空（`embeddingText()` 退化为 `title + breadcrumb`，仅导航文本）
- 浪费 embedding 计算 + 存储

**典型场景**：
```markdown
# 员工考勤与休假管理制度    ← H1，content 空（下一行直接是 H2）
## 1. 总则                  ← H2，有正文
本制度目的...
## 2. 范围                  ← H2，有正文
适用于...
```

**实施结果**：

修改 1 个文件：

**`service/KnowledgeDocService.java`**（修改）
- `flattenDfs`：增加 content 判空守卫，`null / blank` 时跳过该节点（不入 `all` 列表），但**继续递归 children**——子节点照常入库
- `saveChapterNodes`：扁平化后增加 `all.isEmpty()` 兜底判空（过滤后可能所有节点都被跳过），WARN 日志后 return
- `insertNodeRecursive`：**与 flattenDfs 同步**加判空——空 content 节点不生成 chunk 记录，但继续递归 children，`parentId` 透传给子节点（指向最近的祖先非空节点）。修复初版只改 flattenDfs 导致的 NPE：`indexMap.get(node)` 对被跳过节点返回 null → `embeddings.get(null)` 抛 NPE

**范围说明**：
- 不改 `build()`：树结构本身完整，便于后续其他用途（如目录展示、父子注入检索）
- 不聚合空父节点的子节点 content（避免 embedding 信号重复）
- 不为被跳过父节点单独构造"导航 chunk"（breadcrumb 已携带，无价值）

**遗留风险**：
- 极端场景：文档所有节点 content 都为空（罕见，如纯目录 md）→ `saveChapterNodes` 走 WARN return，文档上传"成功但无切片"。日志可追溯
- 重建场景（增量更新分支 C）：新版本被过滤后切片数为 0 时，旧切片已先软删 → 文档暂时无可用切片。属合理代价，后续 #13 状态机可加"空切片告警"
